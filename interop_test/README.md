# interop_test — zero-copy access to a Rust core from a separate PyO3 extension

Question: in `../../rust_python` the core is **C++** (nanobind → Python objects,
autocxx → Rust view); a separate Rust extension borrows those objects zero-copy by
dereferencing a raw pointer obtained from `._addr()`. **Is the same possible when the
core is Rust, and is there a more appropriate mechanism?**

**Answer: yes, and it's cleaner.** This project proves it.

The data structures are **not** redefined here — there is one definition, in the core
crate `rust_core_test` (`../src/lib.rs`). This demo only adds a consumer:

```
(producer)      = the real core: `cr_core` python module, built from ../ with the
                  `python` feature (../src/bindings/python.rs). Its PyPoint/PyState/
                  PyLanelet wrappers expose `_cr_capsule()` for the handoff.
rust_consumer/  = a SEPARATE PyO3 extension. Depends on `cr_core` with the `pyref`
                  feature and borrows producer objects zero-copy via the reusable
                  `cr_core::PointRef`/`StateRef`/`LaneletRef` types. It adds NO struct
                  or FromPyObject plumbing of its own — just its #[pyfunction]s.
cpp_consumer/   = a SEPARATE C++/nanobind extension, split into two layers:
                  - src/logic.{h,cpp}: core logic over the cxx-generated
                    `cr_core::Point/State/Lanelet` — no Python/nanobind awareness.
                  - src/bindings.cpp: the nanobind layer; extracts the capsule
                    pointer and delegates to cr_logic. Borrows the SAME producer
                    objects zero-copy.
main.py         = Rust-core -> Python -> Rust-consumer proof.
cpp_demo.py     = Rust-core -> Python -> C++-consumer proof.
```

So one Rust core, exposed to Python once, is borrowed zero-copy by **both** a Rust and a
C++ downstream extension — through the same `_cr_capsule()` pointer.

Run:

```sh
uv sync --python 3.13
uv run python main.py        # Rust consumer
uv run python cpp_demo.py    # C++ consumer
```

Observed output (both): same address on both sides, a consumer-side `bump_point_x(+10)`
makes Python see `p.x == 11.0`, and passing a `State` where a `Point` is expected is
rejected by the capsule name guard.

## Findings

1. **It works, and needs no autocxx.** With a C++ core, autocxx exists only to give
   Rust a view of the C++ layout. With a Rust core the consumer just depends on the
   `cr_core` crate (`cr_core = { path = "../.." }`) and dereferences to the **real**
   `Point`/`State`/`Lanelet` — the same definitions the producer uses.

2. **Why a pointer handoff is still required.** You cannot just take a
   `PyRef<PyPoint>` in the consumer: a `#[pyclass]` registers its type object in
   per-module static storage, so two separately-compiled extensions get *different*
   Python types and the downcast fails (pyo3 #1444). Sharing the object's memory via a
   pointer is the way across the module boundary.

3. **The more appropriate mechanism: a named `PyCapsule`** (CPython's standard
   primitive for passing C pointers between extensions) instead of the ad-hoc
   `_addr()` int + `_cr_type_tag()` string compare:
   - Producer: `_cr_capsule()` → `PyCapsule_New(&self.0, "cr_core.Point", NULL)`.
   - Consumer: `PyCapsule_GetPointer(cap, "cr_core.Point")` — the **name is the type
     tag**, validated by CPython; a wrong-type capsule returns NULL → raises. No manual
     string comparison, and the type check can't be forgotten.
   - The shared name is a single `&'static CStr` const in `cr_core` (CPython does not
     copy the capsule name, so it must outlive the capsule; sharing the const also
     guarantees producer and consumer agree).

4. **Soundness requires `#[repr(C)]`** on the shared structs — dereferencing a pointer
   to a `#[repr(Rust)]` value produced by a *different* compilation unit is UB. The
   leaf structs use `#[repr(C)]`. Caveat: `Lanelet.left_bound: Vec<Point>` is
   `repr(Rust)` internally, so reading it cross-module is sound only because both
   extensions are built with the same toolchain against this same crate. For a stable
   ABI, expose bounds as a `#[repr(C)]` slice/pointer pair instead of `Vec`.

5. **Lifetime safety** is the same contract as the C++ version: the borrow is tied to
   `'py`; pyo3 keeps the argument object alive for the call, and the borrow checker
   prevents stashing the `&Point` beyond it.

6. **The same handoff works for a C++ consumer (`cpp_consumer/`).** A nanobind module
   reads the *same* capsule and reinterprets the pointer as the **cxx**-generated
   `cr_core::Point`. The producer's pointer is to the Rust `#[repr(C)]` struct; the cxx
   shared struct has identical layout, so the deref is valid — Point/State are pure
   header consumption (no Rust linking; reading `Lanelet`'s `rust::Vec` bounds *would*
   need the cxx runtime, so the demo reads only `id`). The cxx headers are generated by
   `cargo build --features cpp` (driven from `cpp_consumer/CMakeLists.txt`). Net result:
   the C++ side sees the Rust core's objects through the cxx struct view, zero-copy.

## Downstream projects don't re-implement any of this
The borrow glue is generic and lives **once** in `cr_core` behind the `pyref` feature
(`../src/bindings/pyref.rs`): `PointRef`/`StateRef`/`LaneletRef` (+ `…RefMut`), each
`FromPyObject` + `Deref`. Any number of downstream PyO3 extensions just do:

```toml
cr_core = { version = "…", features = ["pyref"] }
```
```rust
use cr_core::PointRef;
#[pyfunction] fn my_fn(p: PointRef) { /* p.x, p.y … */ }
```

So nothing is redefined per project: structs once (cxx bridge), producer bindings once
(`python.rs`), consumer borrow types once (`pyref.rs`). `rust_consumer/` is just an
example that consumes them. Note the two pyo3 features are split on purpose — `pyref`
must NOT pull in the producer's `#[pymodule]`, or a consumer would link a stray
`PyInit_cr_core` into its own extension.

## When you would NOT need any of this
If the consuming code can live in the **same** extension module as the core's pyclasses,
take a native `PyRef<'_, PyPoint>` — zero `unsafe`, fully supported. The capsule/pointer
dance is only needed because the producer and consumer are *separate* compiled modules
(the real-world "downstream package consumes a shared core" case).
